JS 盲點

  1. 除錯必須執行程式
  2. typo
  3. 執行過程沒錯，但結果出乎預期（e.g. 資料格式）
  4. 邊緣情境未被處理

動態語言 v.s. 靜態語言

  • 動態：程式在 執行 當下才會知道型別
  • 靜態：程式的型別在 宣告 時就知道
  • 漸進(TS)：動 + 靜，有宣告的是靜態，沒宣告的靠推論

型別註記 v.s. 型別推論

  • 註記(Type Annotation)：宣告動作
  • 推論(Type Inference)：運送之後決定的型別

強型別 v.s. 弱型別

  • 強：固執、一板一眼
  • 若：有 轉型(Coerce) 的現象

TS 是一門有著 漸近型別 的 弱型別 語言

TS 改善解決的問題、優點

  • 在程式 未執行前 會提示潛在錯誤
  • 較完善的物件導向語法(OOP)
  • 完善定義檔

TS 缺點

  • 命名
  • 型別與介面
  • 第三方套件支援
  • 命名空間與模組

TS 型別系統概論

型別推論與註記

型別推論與註記的使用沒有硬性規定，看情況決定使用比例

  • 函式的參數要標記型別
  • 不直覺、不易推想出來的也建議用註記

型別註記：註記 V.S. 斷言

註記(Annotation)

  • 帶有冒號：系列
  • 寫法：
  1. 一般：(input:number):boolean
  2. 指派型：(input:number) => boolean

斷言(Assertion)

  • 無法推論某 表達式 之確切運算結果之型別 (e.g. 第三方資源、套件、API 結果)
  • 只要是表達式，任何表達式都可以被斷言
  • 斷言不能寫在名稱宣告部分，應該是在被指派的值或表達式的運算結果上
  • 寫法
  1. as
  2. <T>(...)

差異

  • 註記：任何被註記到的變數、函式的參數等，都必須 遵照 被註記過後的變數型別。
  • 斷言：被斷言過後的表達式之運算結果就是某某型別（覆蓋）。

TS 型別種類

  • Primitive Types：資料型別的最小單位概念
  • 物件型別
  • TS 提供型別
    • 元祖 (Tuple)：限制數量 & 順序
    • 列舉 (Enum)：描述相似性質的資料
  • 特殊型別
    • any
    • never
    • unknown
  • 進階型別
    • 泛用型別(Generic)：參數化

明文型別 (Literal Types)

  • 就是「值」的表達方式
  • 除了函式 ，都可以自成一種型別
    const foo:1|2|3 = 1
